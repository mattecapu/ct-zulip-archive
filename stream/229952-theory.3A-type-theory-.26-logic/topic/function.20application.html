For some reason, I didn't like the common way I saw Terms inductively defined in first-order logic, which is sometimes like this:

I am pretty sure that since we said that fff is a function symbol, the notation "f(…)f(\ldots)f(…)" is not defined in this context. fff is an element of a set.

We can associate a function symbol with some arguments, by saying "if fff is a function symbol of arity nnn, and t1,…,tnt_1, \ldots, t_nt1​,…,tn​ are terms, then (f,t1,…,tn)(f, t_1, \ldots, t_n)(f,t1​,…,tn​) is a term".

This still did not feel formal enough for me, and I prefer the following.

Let aaa be a function F→NF \to \mathbb{N}F→N.

For each f∈Ff \in Ff∈F, and for each t∈Ta(f)t \in T^{a(f)}t∈Ta(f), (f,t)∈T(f, t) \in T(f,t)∈T.

That is to say, if nnn is the arity of function symbol fff, take the nnn-ary product of the set of terms TTT, and associate fff with each nnn-ary tuple of terms.

I believe this helps one to move in a more abstract direction where they can look at the elements of first-order logic as well-defined mathematical concepts.

in compiler land, if the parenthesis matter, we would call that "concrete syntax"; if they don't, we would call that "abstract syntax". Concrete syntax only "matters" if we are interested in parsing and printing abstract syntax into strings, where concrete defines things like which strings yield ambiguous parse trees. Sometimes, a syntax with binding constructs (say, untyped typed lambda calculus) is equivalent to one without binding constructs (say, SKI combinatory algebra). So that is one way of avoiding variables in the syntax - instead of writing "\x.x" you write "id", instead of "\x y. x" you write "fst", etc. (Use SKI algebra instead of lambda calculus to avoid variables). The first compilers for ML worked like this, translating away "variables" into higher-order, variable-free, "combinators". That is one way to interpret, "if you want variables, upgrade to higher order".
