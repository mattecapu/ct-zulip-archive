Topic: A parametrized “application” function

It appears this can be generalized to any natural number:

applyn:n-aryFunctionSymbol×Termn→Term\text{apply}_n : \text{n-aryFunctionSymbol} \times \text{Term}^n \to \text{Term}applyn​:n-aryFunctionSymbol×Termn→Term

Perhaps we can make this more formal by adding an equational constraint, for the function symbols. There could be a function select:N→FunctionSymbol\text{select} : \mathbb{N} \to \text{FunctionSymbol}select:N→FunctionSymbol and select(n)={x∈FunctionSymbol∣arity(x)=n}\text{select}(n) = \{ x \in \text{FunctionSymbol} | \text{arity}(x) = n\}select(n)={x∈FunctionSymbol∣arity(x)=n}. (arity:FunctionSymbol→N\text{arity} : \text{FunctionSymbol} \to \mathbb{N}arity:FunctionSymbol→N).

I suppose we could also say that if we have an arity function, select(n)\text{select}(n)select(n) is the pre-image of arity\text{arity}arity at value nnn.

Now it seems like the application function is “fully parametrized”, as applyn:select(n)×Termn→Term\text{apply}_n : \text{select}(n) \times \text{Term}^n \to \text{Term}applyn​:select(n)×Termn→Term. So I think it can be abstracted into Apply:N→{apply}\text{Apply} : \mathbb{N} \to \{ \text{apply} \}Apply:N→{apply}.

I imagine this is a very common, standard function, and I’m wondering what a standard, elegant way to define it is, in contrast to my own attempt.

Ryan Wisnesky (Nov 17 2024 at 03:32):

in languages with products, often all function applications are unary, and if you need n-ary arguments, you tuple them up. or, in e.g. Coq you could define a family of functions of varying arity by induction on the number of arguments.
