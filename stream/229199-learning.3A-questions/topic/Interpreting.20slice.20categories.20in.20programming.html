Topic: Interpreting slice categories in programming

Davi Sales Barreira (Dec 08 2023 at 15:02):

First let me give some context. I'm coding an application, and I'm trying to interpret within category theory. Thus, I've started by assuming that the programming code was in the category of sets, with types as sets, and programming functions as mathematical functions in set-theory.

From that, I've coded a free monad D:Set→SetD:\textbf{Set}\to \textbf{Set}D:Set→Set. Given a type XXX, an element of DXDXDX is an expression tree where the leafs have values of type XXX.

In my code, I have an special type called PPP, which represents primitives. Given a type (set) AAA, I wanted to assign a function θA:A→DP\theta_A: A \to D PθA​:A→DP, i.e. a function that receives a value of type AAA and constructs an expression formed by primitives. After some discussion, some people pointed out that the whole thing could be seen as the slice category Kleisli(D)/P\text{Kleisli}(D)/PKleisli(D)/P. Indeed, after inspection, they were right, and this interpretation fits nicely (I wanted the composition to work like it does in the Kleisli category).

Here is my new conundrum. I now want to use my monad DDD to construct expression trees where the leafs have "type" Kleisli(D)/P\text{Kleisli}(D)/PKleisli(D)/P, but Kleisli(D)/P\text{Kleisli}(D)/PKleisli(D)/P is a category not a set. Hence, I can't just interpret it as a type. I was wondering if there is a way to interpret this category in programming. My first idea was to somehow represent Kleisli(D)/P\text{Kleisli}(D)/PKleisli(D)/P as a set or perhaps as a subcategory within the category of sets. I thought of Yoneda lemma (which I am frankly still in the process of fully understanding).
